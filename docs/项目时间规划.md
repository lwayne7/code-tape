# 代码讲解工具两周冲刺时间规划

> **阅读摘要**
>
> 本文基于 [技术模块拆解](技术模块拆解.md) 制定两周冲刺计划。当前 DDL 调整为 **2026-06-03**，因此项目目标需要收缩为「两周内完成 P0 主链路，并交付至少一个可展示加分项」。P0 基本项覆盖 JavaScript / TypeScript；其他语言支持作为加分项处理。技术方案阶段仍负责敲定代码执行路线、TypeScript 执行深度、录制包保存方式等关键问题。

## 一、目标与范围

### 1. 时间目标

以 2026-05-22 启动计算，DDL 为 2026-06-03。实际可用开发窗口约 12 天，需要采用「主链路优先、加分项最小可展示、最后一天冻结」的策略。

阶段比例调整为：

- 20% 时间用于技术方案、关键选型和高风险 PoC。
- 50% 时间用于 P0 录制与回放主链路。
- 15% 时间用于加分项最小展示。
- 15% 时间用于联调、稳定性、Demo 准备和兜底。

### 2. P0 必达范围

P0 必须围绕 [技术模块拆解](技术模块拆解.md) 中的五个主流程模块完成闭环：

| 模块 | P0 要求 |
| --- | --- |
| 代码编辑器模块 | 支持 JS / TS 编辑体验、语法高亮、基础补全、工具栏、代码运行入口 |
| 操作录制模块 | 采集内容变更、光标选区、鼠标轨迹、快捷键，并生成带时间戳事件流 |
| 音视频录制模块 | 采集麦克风、摄像头，支持基础开关和摄像头预览 |
| 录制总控模块 | 支持开始、暂停、继续、结束录制，生成统一录制包 |
| 回放模块 | 支持播放、暂停、倍速、seek、音量控制，并同步还原编辑器、鼠标、选区、快捷键和摄像头 |

P0 演示验收口径：

1. 用户可以在编辑器中完成一段 JS / TS 代码讲解。
2. 系统可以录制编辑器内容变化、光标选区、鼠标轨迹、快捷键、麦克风和摄像头。
3. 录制结束后可以生成录制包。
4. 回放时可以按时间轴还原讲解过程。
5. 回放支持暂停、继续、倍速、进度条 seek、音量调节和静音。

### 3. 加分项范围

两周排期下，加分项必须按「最小可展示」处理，不能追求完整产品化。

| 加分项 | 优先级 | 两周内目标 |
| --- | --- | --- |
| 其他语言支持 | 必做加分项 | 除 JS / TS 外，至少支持 1 种语言的高亮、切换和回放恢复 |
| AI 字幕基础版 | 备选加分项 | 优先打通字幕回放和点击 seek；字幕来源可以先用模拟字幕或离线生成结果 |
| WebRTC 实时面试最小链路 | 冲刺加分项 | 只在 P0 稳定后尝试；优先实时同步编辑器事件，不强求完整双向音视频 |

本计划建议：**其他语言支持作为必做加分项，AI 字幕基础版作为次优先加分项，WebRTC 作为冲刺展示项。**

## 二、核心推进原则

### 1. P0 主链路不可被加分项打断

两周内最重要的技术主线是：

```text
编辑器集成
-> 操作事件采集
-> 音视频采集
-> 录制总控打包
-> 回放时间轴调度
-> 同步展示
```

任何加分项如果需要改动这条主链路，必须在技术方案阶段明确接口，不能在实现后期临时侵入。

### 2. 技术方案要做决策，不写大而全

技术方案阶段必须在 5/24 前回答这些问题：

- 代码执行路线选择后端代码沙箱，还是前端页面展示。
- TypeScript 是 P0 编译执行，还是 P0 只要求编辑体验和展示。
- 音频和摄像头是否合并为同一个媒体文件。
- P0 录制包只做本地保存，还是必须接入上传。
- 其他语言加分项第一批支持哪一种语言。

这些问题如果 5/24 仍未确定，默认按低风险方案降级，保证 P0 主链路继续推进。

### 3. 加分项提前留口，但后置接入

第一阶段需要预留这些扩展点：

- 事件模型保留 `language`、`timestamp`、`source`、`track` 字段。
- 录制包保留字幕、章节、多语言元数据扩展位。
- 回放模块预留字幕层和实时事件输入层。
- 编辑器模块预留语言注册和语言切换机制。

但真正的加分项实现应集中在 P0 录制与回放链路基本可用之后，避免主线失焦。

阶段一要在 3 天内完成「调研 + 选型 + PoC + 架构冻结」。这不是资料收集阶段，而是为后续实现扫雷。结束后，组员应当可以按 Issue 直接进入开发。

| `[PoC] 回放调度与 seek 策略` | PoC | `score:5` | `stack:typescript` | 播放、暂停、倍速、seek 的技术验证 |

| `[加分项] AI 字幕最小方案` | 方案/PoC | `score:2` | `stack:research` | 字幕文件格式、字幕回放、点击 seek 方案 |

### 3. 调研文档要求

每个调研或 PoC Issue 必须产出 Markdown 文档，建议放在 `docs/research/` 下。文档必须包含：

```markdown
# 标题

## 1. 对应模块

说明本调研服务于哪个技术模块。

## 2. 要回答的问题

列出 3-5 个必须回答的问题。

## 3. 调研对象或验证方案

列出产品、库、API、技术路线或 PoC 验证方式。

## 4. 核心结论

只写对 P0 或加分项有决策价值的结论。

## 5. 方案对比

## 6. 推荐方案

## 7. 风险与待验证点

列出仍需在实现阶段关注的问题。

## 8. 后续 Issue 建议

把调研结论转成可以继续认领的实现任务。
```

## 五、阶段二：P0 录制链路

### 1. 阶段目标

阶段二要完成从「打开编辑器」到「生成录制包」的闭环。此时回放可以还不完整，但录制数据必须结构稳定，否则阶段三会被迫返工。

### 2. 推荐 Issue 拆分

| Issue 标题 | 依赖 | 建议分数 | 交付物 |
| --- | --- | --- | --- |
| `[P0] 编辑器基础集成` | 编辑器选型 | `score:5` | 编辑器组件、JS / TS 高亮、基础补全 |
| `[P0] 编辑器工具栏与语言设置` | 编辑器基础集成 | `score:3` | 语言切换、字号设置、常用操作入口 |
| `[P0] 内容变更事件采集` | 事件模型 | `score:3` | `content-change` 事件采集 |
| `[P0] 光标与选区事件采集` | 事件模型、编辑器集成 | `score:3` | `selection-change` 事件采集 |
| `[P0] 鼠标轨迹采集与降频` | 事件模型 | `score:3` | `mouse-move` 事件采集和采样策略 |
| `[P0] 快捷键事件采集` | 事件模型、编辑器集成 | `score:3` | `shortcut` 事件采集和展示 label |
| `[P0] 麦克风与摄像头采集` | 音视频 PoC | `score:5` | MediaStream 管理、权限处理 |

### 3. 阶段验收标准

- 可以启动一次录制。
- 可以暂停、继续、结束录制。
- 录制过程中可以看到录制时长。
- 事件流中包含内容变更、光标选区、鼠标轨迹、快捷键。
- 音视频数据可以被采集并写入录制包。
- 录制包包含 JS / TS 语言信息。

- 摄像头视频和操作事件基本同步。

## 七、阶段四：加分项最小展示与体验补齐

### 1. 阶段目标

阶段四只有 1 天，不能做完整加分项产品化。目标是把已经预留好的扩展点展示出来，形成评审时能看懂的亮点。

### 2. 加分项推荐顺序

| 顺序 | 加分项 | 两周内做法 |
| --- | --- | --- |
| 1 | 其他语言支持 | 至少增加 1 种语言高亮和语言切换，录制包记录语言，回放恢复语言 |
| 2 | AI 字幕基础版 | 用模拟字幕或离线字幕文件先打通字幕展示、滚动和点击 seek |
| 3 | WebRTC 实时面试最小链路 | 仅当 P0 已稳定时尝试实时事件同步，不追求完整音视频通话 |

### 3. 推荐 Issue 拆分

| Issue 标题 | 类型 | 建议分数 | 交付物 |
| --- | --- | --- | --- |
| `[加分项] 其他语言高亮与语言切换扩展` | 必做 | `score:3` | 除 JS / TS 外至少支持 1 种语言高亮 |
| `[加分项] 多语言录制包元数据支持` | 必做 | `score:2` | 录制包记录语言信息，回放时恢复语言状态 |
| `[加分项] 字幕回放与点击 seek` | 备选 | `score:3` | 字幕随播放进度滚动，点击字幕跳转 |
| `[加分项] AI 字幕生成基础链路` | 备选 | `score:3` | 从音频生成字幕，若模型接入风险高则使用离线字幕结果 |
| `[加分项] WebRTC 实时事件同步最小链路` | 冲刺 | `score:5` | 被面试者事件实时同步到面试官端 |
| `[P1] UI 状态反馈与展示开关` | 体验补齐 | `score:3` | 录制/暂停/保存/回放状态反馈，快捷键/鼠标/字幕展示开关 |

不建议在两周版中单独开 WebRTC 双向音视频通话 Issue。这个任务风险太高，容易吞掉主链路稳定时间。

### 4. 加分项验收标准

必做加分项至少满足：

- 除 JS / TS 外，至少一种语言可以切换并正确高亮。
- 录制包可以记录当前语言。
- 回放时可以恢复录制时的语言状态。

备选加分项满足其一即可：

- 字幕可以在回放页面展示，并随时间轴变化。
- 点击字幕可以 seek 到对应时间点。
- 实时事件同步可以在两个页面之间展示编辑器状态变化。

## 九、并行分工建议

如果组员人数足够，可以按技术线并行：

| 小组 | 负责范围 | 关键协作对象 |

- 事件 schema。
- 录制包格式。

- 媒体时间轴对齐规则。

## 十一、风险与兜底策略

| 风险 | 影响 | 兜底策略 |
| --- | --- | --- |
| 两周时间过短 | P0 和加分项互相抢时间 | P0 主链路优先，加分项只做最小展示 |
| seek 状态恢复复杂 | 回放体验不稳定 | 第一阶段必须做 PoC；实现时优先支持最近状态恢复 |
| 音视频与事件流不同步 | 讲解回放观感差 | 统一使用录制起点时间轴，媒体和事件都记录相对时间 |
| TypeScript 执行成本高 | 拖慢 P0 | 技术方案阶段明确是否只做 TS 编辑体验；执行能力可降级 |
| 后端沙箱安全复杂 | 实现风险高 | 可先采用前端 iframe 展示或受限执行作为 P0 路线 |
| 设备切换逻辑复杂 | 录制模块延期 | P0 优先默认设备采集，设备枚举和切换后置 |
| WebRTC 风险高 | 加分项不可控 | 降级为实时事件同步，不强求完整双向音视频 |
| AI 字幕模型接入不稳定 | 加分项延期 | 允许先用模拟字幕文件打通字幕回放和 seek |
| 最后一天新增功能 | Demo 不稳定 | 6/2 晚上冻结新增能力，6/3 只做联调和演示打磨 |
