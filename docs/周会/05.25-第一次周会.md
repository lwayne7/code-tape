# 5.25 第一次周会：P0 主链路收口与 Demo 验收

## 0. 会前阅读摘要

当前项目已经从调研和技术方案阶段进入 P0 主链路实现阶段。到 2026-05-25 会前为止：

- 已完成竞品调研、P0 技术方案、核心 ADR 决策沉淀。
- Monaco CodeEditor 基础集成已合并，编辑器选型和工程接入已经落地。
- 当前正在并行推进 RuntimeProducer、RecorderControls、ReplayControls、RecordingLibraryPage、CameraPreview 等 P0 任务。
- DDL 是 2026-06-03，剩余窗口很短，本次会议应聚焦“是否继续收口”和“如何保证 Demo 可验收”。

本次会议不再逐条复述 [PRD](../PRD/代码讲解工具.md) 和 [技术方案](../技术方案.md)。我希望导师重点确认 5 件事：

1. **P0 边界**：主链路优先、加分项最小化的收敛是否被认可。
2. **技术路线**：事件流、录制包、iframe sandbox、seek、媒体同步是否有明显硬伤。
3. **Demo 验收**：最终展示应突出“结构化回放体验”，还是突出“在线运行代码效果”。
4. **加分项取舍**：第一加分项是否选 Python 高亮/语言切换/回放恢复。
5. **推进节奏**：是否需要从 5/26 开始固定端到端联调，避免月底集中爆雷。

## 1. 需要导师拍板的结论

### 1.1 P0 范围是否继续收口

当前 P0 的核心判断是：**先证明“结构化事件流录制 + 可交互回放”成立，而不是做完整在线 IDE、判题平台或云端课程平台。**

| 能力 | 当前口径 | 需要导师确认 |
| --- | --- | --- |
| 代码编辑与录制 | JS/TS 编辑、高亮、基础补全、内容/选区/鼠标/快捷键事件采集是 P0 必达 | 是否还缺少必须补上的录制事件 |
| 代码运行 | P0 采用前端 iframe sandbox 展示 JS 运行结果；TS 简单运行只作为可选 PoC | 是否认可不做完整后端代码沙箱 |
| 录制包与保存 | P0 优先 IndexedDB 本地保存，zip 导入导出作为兜底 | zip 是否必须在 Demo 中显性展示 |
| 回放能力 | 播放/暂停/倍速/seek/音量/静音，以及编辑器、鼠标、选区、快捷键、摄像头同步回放是 P0 必达 | 是否认可这是主验收口径 |
| UI 与主题 | 深浅主题按技术方案进入 P0，但只作为基础体验，不抢主链路优先级 | 是否需要在答辩中特别展示 |
| 云端、WebRTC、AI 字幕、多文件、终端、npm | 全部延后到 P1/P1+；P0 只保留 schema 或接口余量 | 是否认可当前降级策略 |

### 1.2 技术路线是否有硬伤

P0 技术路线一句话概括：

> 用结构化事件流作为事实源，音视频作为同步轨道；录制阶段生成本地录制包，回放阶段用统一时间轴恢复编辑器、运行结果、媒体和控制状态。

| 决策 | 当前方案 | 会上只需要确认的问题 |
| --- | --- | --- |
| 编辑器 | Monaco Editor | 是否认可继续围绕 Monaco 暴露的内容、选区、快捷键能力实现 |
| 运行路线 | iframe sandbox + postMessage + runId/source/schema/timeout 校验 | 如果被问“不做后端沙箱”，用 PRD 二选一和 P0 风险收敛解释是否充分 |
| 内容变更事件 | P0 保存变更后的完整代码快照 | 演示规模下体积是否可接受 |
| 录制包 | meta、events、snapshots、media、manifest；IndexedDB 保存 | manifest/checksum/zip 兜底是否足够 |
| seek | 最近 inclusive 快照 + 增量事件静默重放 | 是否需要在 Demo 前增加专项压测 |
| 媒体同步 | 麦克风和摄像头合并为单 WebM，通过 MediaClockAdapter 映射到事实时间轴 | 媒体失败时是否允许用备用包兜底 |
| 回放运行结果 | 展示历史运行结果，不默认重新执行代码 | 是否认可用确定性换取安全和稳定 |

详细设计已沉淀在 [技术方案](../技术方案.md) 和 ADR，本次周会只讨论是否需要改方向。

### 1.3 Demo 验收口径是否准确

建议 Demo 的主叙事是：

> code-tape 不是录屏工具，而是代码讲解的结构化时间线。它能把写代码、讲解、运行、选区、鼠标、快捷键、摄像头组织成可 seek、可回放、可扩展的录制包。

建议展示闭环：

1. 打开录制工作台，在 Monaco 中讲解一段 JS/TS 代码。
2. 触发运行，iframe sandbox 输出结果，并把结果写入事件流。
3. 开始录制，展示计时、暂停/继续、麦克风/摄像头开关和摄像头预览。
4. 停止录制，生成录制包并保存到本地回放中心。
5. 从回放中心进入回放页，展示代码变化、运行结果、鼠标激光笔、光标选区、快捷键浮层、摄像头视频。
6. 演示播放/暂停、0.5x/1x/1.5x/2x、进度条 seek、音量和静音。

建议演示约束：

| 项目 | 建议值 |
| --- | --- |
| 演示环境 | 桌面 Chrome/Edge latest |
| 单次录制 | 5 分钟以内 |
| 单文件代码 | 200KB 以内 |
| previewHtml | 200KB 以内 |
| 事件数量 | 20,000 条以内 |
| media Blob | 200MB 以内 |
| seek 恢复目标 | 300ms 内恢复稳定编辑器状态 |
| 媒体漂移目标 | 小于 250ms |

需要导师确认：**最终展示时更应该强调“结构化回放体验”，还是强调“在线运行代码效果”？**
我当前判断是前者，因为 PRD 的差异化核心是录制与回放引擎，不是代码执行环境。

### 1.4 加分项优先级是否认可

结合 [项目时间规划](../项目时间规划.md)，两周内至少需要一个可展示加分项，但不能打断 P0 主链路。

| 加分项 | 当前建议 | 原因 | 边界 |
| --- | --- | --- | --- |
| Python 高亮/语言切换/回放恢复 | P0 主链路稳定后优先做 | 成本低，能证明多语言录制扩展能力 | 不做 Python 执行、包管理、后端沙箱 |
| AI 字幕基础 UI 或模拟字幕回放 | P0 稳定后再评估 | 展示价值高，但 ASR、时间戳、术语纠错风险大 | 不作为 P0 主答辩承诺 |
| WebRTC 实时事件同步 PoC | 只作为冲刺展示项 | 网络同步、多人身份、音视频通话复杂度高 | 不抢单人录制回放主链路 |

需要导师确认：**第一加分项是否选 Python 高亮/语言切换/回放恢复，而不是 AI 字幕或 WebRTC？**

### 2.2 当前判断

- **方向健康**：当前 issue 基本都在 P0 主链路上，没有明显偏到 P1/P1+。
- **最大风险是集成**：单个模块边界清晰，但录制和回放最终会在页面层、事件流、录制包和调度器处汇合。

## 4. 风险清单与会上问题

| 风险 | 当前缓解 | 希望导师判断 |
| --- | --- | --- |
| P0 范围继续膨胀 | ADR 已明确延后后端沙箱、WebContainers、WebRTC、AI 字幕、云端分享 | 是否还需要继续砍范围 |
| seek 状态恢复不稳定 | inclusive 快照 + 增量静默重放 | 是否需要提前做专项压测 |
| 媒体同步复杂 | 单 WebM + MediaClockAdapter | 媒体失败时是否允许备用包兜底 |
| iframe runtime 安全边界不足 | source/runId/schema/timeout 校验 | 是否还需要补充限制 |
| IndexedDB quota 或导入导出失败 | manifest/checksum/zip 兜底 | zip 是否必须进入主 Demo |
| 端到端联调太晚 | 建议从 5/26 开始每日集成检查 | 是否指定集成负责人 |

会上想问导师的问题按优先级排序：

1. 当前 P0 范围是否合理？有没有应该立即砍掉或必须补上的能力？
2. Demo 是否应突出“结构化时间线”，而不是“在线 IDE”？
3. iframe sandbox 作为 P0 代码运行路线是否能通过验收？
4. 完整代码快照作为 P0 content-change 事件是否可以接受？
5. seek 的“inclusive 快照 + 增量事件静默重放”是否足以支撑 P0？
6. 媒体同步是否必须现场实时演示，还是可以用备用包兜底？
7. P0 演示环境锁定 Chrome/Edge latest 是否可接受？
8. Python 高亮是否是最合适的第一加分项？
9. 团队是否需要更早端到端联调？
10. 答辩时如何解释“这不是录屏工具”的差异化价值？
