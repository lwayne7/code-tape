# GitNexus 代码图谱契约

本文说明 code-tape 的代码图谱契约：本地 agent 先用 GitNexus 建立代码图谱并做影响分析，PR 再由 CI 阻断兜底。本仓库不再接入外部知识库服务，权威上下文以仓库内文档为准。

## 本地 agent 工作流

所有 agent 在改代码前先运行：

```bash
npm run agent:bootstrap
npm run quality:predev
```

提交时通过 git hook 自动运行快速质量闸门：

```bash
git commit
```

推送时通过 git hook 自动运行完整质量闸门：

```bash
git push
```

不要在正常提交/推送流程中手动先跑 `npm run quality:precommit` 或 `npm run quality:local`；这些命令已经分别归 `pre-commit` 和 `pre-push` hook 所有。只有在诊断 hook 失败、做本地预检、明确使用 `SKIP_QUALITY_HOOKS=1` 绕过 hook，或当前环境没有安装/触发仓库 hook 时，才需要手动运行对应质量门。

`contract:local` 会强制执行 `npx --yes gitnexus@1.6.5 analyze --force --index-only`，只刷新 `.gitnexus/` 索引，不注入或重写 `AGENTS.md` / `CLAUDE.md`。本地默认 60 秒超时，CI 默认 180 秒超时；如仓库确实需要更长时间，可设置 `GITNEXUS_ANALYZE_TIMEOUT_MS`。超时会让命令失败，避免 agent 继续依赖陈旧索引。

随后脚本检查当前 diff 是否触碰关键骨架，包含新增、修改、重命名和删除。命中以下区域时，必须阅读脚本输出的 GitNexus 建议，并在 PR 的 “GitNexus 影响分析摘要” 中写明结论：

- `apps/web/src/shared/recording-schema/`
- `apps/web/src/features/runtime-preview/`
- `apps/web/src/features/library/recordingStore*`
- `apps/web/src/features/player/replay*`、`packageLoader*`、`timelineClock*`、`mediaClockAdapter*`
- `.github/workflows/`、`scripts/workflows/`
- `README.md`、`AGENTS.md`、`CLAUDE.md`、PRD、技术方案、流程文档

本仓库使用原生 git hooks 作为质量闸门基础设施。`npm install`、`npm ci` 或 `npm run agent:bootstrap` 会安装 `.githooks`；`pre-commit` 运行 `quality:precommit`，`pre-push` 运行 `quality:local`。维护者紧急处理时可设置 `SKIP_QUALITY_HOOKS=1` 跳过本地 hook，但 CI 不接受该绕过。

## PR / CI 契约

PR 阶段会运行：

```bash
npm run contract:gitnexus
```

`contract:gitnexus` 同样强制刷新 GitNexus 索引。关键骨架变更必须同时满足：

- 有匹配的契约测试或 workflow 测试变更。
- PR 模板中填写结构化 GitNexus 影响分析摘要。

摘要必须使用以下字段，且不能保留 `-`、`无`、`todo`、`n/a` 等占位值：

```md
- 风险等级: LOW|MEDIUM|HIGH|CRITICAL
- 关键骨架变更: 说明触碰的关键目录或文件
- GitNexus 影响面: 写明 detect_changes，并至少写明 query、context 或 impact 中一个工具的结论
- 验证结果: 写明已运行的测试命令和结果，或无法运行的原因
```
